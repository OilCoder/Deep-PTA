## 12:30 — Experimento: ¿no-unicidad física o pipeline sin pulir?

### Cambios técnicos
- `debug/dbg_confusion_concentration.py` — nuevo script aislado: carga los dos checkpoints,
  regenera las matrices de confusión crudas sobre `data/synthetic_test.h5` (vía
  `train_cnn.evaluate`) y mide, por clase verdadera, la concentración de errores en pares
  físicamente no-únicos vs un baseline uniforme (`ratio`).
- `documentation/reporte-no-unicidad-confusiones.md` — informe con método, resultados crudos
  y veredicto.

### Decisiones de diseño
- La métrica clave es `ratio = %err→ambig / baseline`: separa no-unicidad real (`ratio ≫ 1`)
  de error difuso por pipeline (`ratio ≈ 1`). Convierte una angustia ("¿nos engañamos?")
  en una pregunta medible.
- Los pares ambiguos se declaran como **conocimiento de dominio explícito** en el script
  (homogéneo↔fractura temprana, infinito↔fronteras no desarrolladas, etc.), no se infieren.
- Script en `debug/` (aislado, gitignored) porque es investigación, no parte del pipeline.

### Resultados (números crudos, test set 1200)
- **Frontera:** el test set es **70% "infinito" (836/1200)** por el filtro de validez →
  el 0.84 está inflado; el promedio balanceado ronda 0.61. Errores con `ratio ≈ 2.8–3.0`
  (casi todo va a "infinito"): coherente con la física, pero confundido con el desbalance.
- **Yacimiento:** errores **difusos**, `ratio ≈ 1` (CNN), <1 en Transformer. Señal física
  débil (fin↔inf fracture `ratio` 1.47; homogéneo 1.22). El Transformer sobre-predice
  fin_fracture (sesgo de modelo, no no-unicidad).

### Conclusión
- **No** se demostró que el DL no pueda con PTA. Frontera = funciona pero métrica engañosa
  por desbalance (arreglable). Yacimiento = under-polished (3000 steps, sin HPO finalizado),
  con piso pequeño de no-unicidad real en los bordes (arreglable). Techo de literatura: 85–95%.

## 15:00 — Plan nuevo + ciclo de mejora de accuracy (Fase 6, E0–E6)

Tras el experimento, el usuario pidió un plan para llevar el proyecto a buen término
(recursos extra y plazos OK). Plan mode + 3 Explore + 1 Plan agent → plan aprobado.
Decisiones: profundizar el MVP (4×4, sin ampliar), validación **solo sintética** con test
estratificado, objetivo portafolio + rigor. Datos reales = trabajo futuro (infra lista).

### Cambios técnicos
- E0: `train_cnn.evaluate()` + `_class_metrics()` → balanced accuracy, macro-F1, recall y
  soporte por clase.
- E1: `sampling.py` muestreo condicionado por observabilidad (`t_max` ~ `L_D²`/`r_eD²`,
  18% ventana corta; etiquetable solo si la firma cae tras storage y dentro de ventana);
  `losses.py` class weights opcionales; `generator.export_stratified_set` (+`split`).
- E2: `representation.py` tercer canal de tiempo absoluto (2→3, normalización fija);
  `resnet1d`/`patchtst` in_channels 3 + dropout; `fit()` AdamW+warmup/cosine+grad clip+
  early stopping; ventana condicionada a `C_D`.
- E3: `hpo.py` objetivo balanced accuracy, MedianPruner, SQLite, eval en val, `best_config`.
- E4: split de validación (banda C_D disjunta), `config.TrainConfig`, checkpoint-best-on-val,
  TensorBoard gated, entrypoint `train/run.py` + `__main__.py`, train congelado opcional.
- E5: test estratificado 3-canal (1200), val (640), train congelado (25k); ablation.

### Decisiones de diseño
- El tercer canal NO se estandariza por curva (eso borraría la posición temporal absoluta);
  se normaliza con constantes globales fijas. Es justo el contexto que faltaba para separar
  storage temprano de firma tardía.
- Train congelado (25k) para desacoplar la generación (CPU, ~20 ms/curva) del entrenamiento
  (GPU). Hace viable el HPO; cuesta diversidad de aumentación.
- Cambios de distribución (muestreo + canal) documentados en docstrings como currículo por
  dominio, mismo condicionamiento en train/val/test; el split disjunto por `C_D` se mantiene.

### Resultados (test estratificado balanceado, 300/clase)
- Frontera: balanced 0.61 → **0.698**; sellante 0.35→0.59, presión cte 0.40→0.60, cerrado
  0.71→0.85; infinito 0.98→0.76 (honesto, ya no sobre-predice). Ratios de concentración 2.0–2.3.
- Yacimiento: balanced 0.43 → **0.51**; homogéneo 0.12→0.33. Ratios 1.0–1.26 (aún difuso).
- HPO (10 trials, best val 0.606) **no superó** la config por defecto en test → el techo del
  yacimiento lo limita la diversidad del train congelado, no los hiperparámetros.

### Errores
- Reentrené con el "best config" del HPO y sobrescribí `cnn_best.pt`, pero dio yacimiento
  peor (0.484 vs 0.513): el HPO optimizó en val (frozen) y no generalizó mejor al test.
  Cómo lo noté: comparar test del run por defecto vs el del HPO. Lo correcto: reentrenar la
  config por defecto para que el modelo enviado coincida con la cifra titular, y reportar
  ambas filas en la tabla de ablation (HPO no mejora = hallazgo, no fracaso).
- El matcher `PreToolUse` siguió dando falsos positivos (`rm -rf` real bloqueado bien, pero
  `git init`/`for`/`&&` también disparaban `force-push`/`--no-verify`). Trabajé alrededor con
  comandos simples; queda como deuda de configuración del hook.

## Aprendizajes
- Una accuracy agregada alta puede ser un espejismo de desbalance: 0.84 de frontera era 70%
  de clase mayoritaria. Siempre mirar soporte por clase y accuracy balanceada.
- "Errores difusos vs concentrados" es un test barato y potente para distinguir un modelo
  que aprendió física de uno que adivina — antes de invertir en más entrenamiento.
- Un hook no sustituye la redacción de la bitácora: enforce ≠ narrativa. La consistencia
  documental es un hábito del operador, el hook solo lo dispara.
- La estandarización per-curva da invariancia de escala pero **borra** la posición temporal
  absoluta; si la firma depende de "dónde en el tiempo" ocurre (storage vs frontera), hace
  falta un canal con normalización global.
- Cuando el HPO no mueve la aguja, el cuello de botella suele ser los **datos**, no los
  hiperparámetros. El HPO sirvió para *descartar* la hipótesis de hiperparámetros.
