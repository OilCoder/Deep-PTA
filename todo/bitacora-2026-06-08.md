## Acciones

- **Split de `C_D` híbrido (E8).** Cambié `generator.split_of` de bandas disjuntas a un
  **holdout i.i.d. por hash de muestra** sobre el rango in-distribution, reservando la banda
  alta (`log10 C_D ≥ 3.5`) como **test de extrapolación**. Regeneré val / test in-dist /
  test extrapolación. Verifiqué no-fuga (0 colisiones de split).
- **Motor GPU (gpu_engine.py).** Porté el motor analítico a PyTorch batched (Stehfest + Bessel
  en GPU, agrupando por clase). Implementé `ive` (Bessel I escalado) que torch no trae:
  exacto para argumento chico, asintótico para grande (frontera cerrada con `r_eD·x` ~87000).
  **Validado contra el motor CPU certificado a 4e-7** en los 16 combos. Test de equivalencia.
- **Escala de datos.** Generador paralelo CPU (2M, 22 min) y generador GPU (motor GPU + post
  CPU, distribución idéntica). Generé **2M** y **5M** congelados, reservoir balanceado.
- **Entrenamientos.** ResNet32 sobre 2M (yac 0.618) y 5M (0.638); ResNet64 grande sobre 5M
  (≈0.633, capacidad aplanada); PatchTST sobre 5M (0.639). **Ensemble final ResNet32 + PatchTST
  (arquitecturas diversas) = yac 0.649 / front 0.765 / MAE 0.357** in-distribution (mejor que
  el ensemble de dos ResNet, 0.644).
- **Cierre.** Tabla de ablations v2, reporte, README, GitHub Pages, métricas y figuras.

## Decisiones de diseño

- **Híbrido sobre disjunto (decisión del usuario):** la métrica de cabecera mide
  in-distribution (sube fuerte) y se reporta la extrapolación aparte (honesto). El `0.513` no
  es comparación directa porque cambió la distribución del test — documentado.
- **Motor CPU sigue siendo la referencia certificada;** el GPU solo acelera la generación. La
  equivalencia 4e-7 está 5 órdenes bajo el ruido de realismo, así que la certificación se
  preserva sin re-certificar contra AnaFlow.
- **Opción A (motor GPU + post CPU)** sobre full-GPU: preserva la distribución y el pilar
  sim-to-real; el full-GPU habría requerido rediseñar el realismo (descartado).
- **No más entrenamientos marginales:** datos y capacidad se aplanaron; el techo es la
  no-unicidad homogéneo↔inf-fractura.

## Errores

- **OOM al entrenar sobre 5M con workers.** Lancé el 5M con `--workers 3`. Murió por OOM sin
  traceback (SIGKILL del kernel). Causa: `FrozenH5Dataset` carga los 15.4GB en RAM y, con
  `shuffle`, cada worker lee índices que abarcan todo el array → el copy-on-write por
  refcounting de Python duplica casi todas las páginas → revienta los 29GB. Lo noté porque el
  proceso desapareció y la RAM volvió a 3GB. Solución correcta: `num_workers=0` (sin fork, sin
  COW). Aprendizaje: datasets grandes en RAM + DataLoader workers = trampa de COW; usar 0
  workers o tensores en memoria compartida.
- **`pkill -f "<patrón>"` se auto-mató.** Para relanzar puse `pkill -f "e_5m_long"` en el mismo
  comando cuyo `argv` contenía "e_5m_long" → pkill mató el propio shell antes de lanzar el
  python (exit 144), así que el entrenamiento nunca arrancó. Lo noté porque RAM seguía en 3GB
  y no había output. Aprendizaje: nunca poner `pkill -f` con un patrón que aparece en la propia
  línea de comando; matar por PID o con un patrón que no se solape.
- **Umbral de validación demasiado estricto.** Puse 1e-8 como criterio de equivalencia GPU↔CPU;
  el real fue 4.3e-7 (numérico torch-vs-scipy), marcado "FAIL" sin serlo. Corregí el umbral a
  1e-5 (sigue muy por debajo del ruido). Aprendizaje: fijar tolerancias contra el piso de ruido
  físico, no contra cero-máquina.
- **Hook de `.claude` con falsos positivos** (bloqueó comandos `ps`/`until` como si fueran
  `--no-verify`). Sorteado simplificando comandos. Deuda pendiente de arreglar el matcher.

## Aprendizajes

- **El HPO nulo de v1 tenía razón:** el cuello era diversidad de datos. 25k→2M→5M subió el
  yacimiento honesto 0.51→0.64 y bajó el MAE 0.69→0.37. Capacidad ≈ neutra. Datos = lever real.
- **GPU para generar, no solo para entrenar:** portar el motor (79% del tiempo de generación) a
  GPU validándolo contra el certificado dio millones de curvas en minutos sin perder rigor.
- La curva de escala se aplana hacia 5M → el siguiente lever es físico (no-unicidad), no más datos.
